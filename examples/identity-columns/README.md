# Identity Columns

<em>The SQL-standard way to auto-increment, without `serial`'s implicit sequences.</em>

`serial` is a PostgreSQL-ism that quietly creates a sequence behind your back. `GENERATED AS IDENTITY` (SQL:2003, PostgreSQL 10+) does the same job in the open — and the `ALWAYS` flavor refuses hand-written IDs entirely, so nobody inserts `id = 42` by accident. This example runs both flavors through a full insert/select/update/count cycle.

## Run

```bash
cd examples/identity-columns
go mod tidy

export DATABASE_URL="postgres://postgres:postgres@localhost:5432/pebble_test?sslmode=disable"

pebble generate --name init_identity_tables --models ./internal/models --verbose
pebble migrate up --all --db $DATABASE_URL

go run ./cmd/identity-example
```

`DATABASE_URL` is optional — the example falls back to the URL above.

## What it shows

| Tag | Generated SQL | When |
|-----|---------------|------|
| `identity` | `GENERATED ALWAYS AS IDENTITY` | Default choice — strict, no manual IDs |
| `identityAlways` | `GENERATED ALWAYS AS IDENTITY` | Same as `identity`, spelled out |
| `identityByDefault` | `GENERATED BY DEFAULT AS IDENTITY` | Auto by default, manual override allowed (data migrations, imports) |

Two models, one of each flavor ([internal/models/models.go](internal/models/models.go)):

```go
// Product: ID is always auto-generated — inserting an explicit ID is an error.
type Product struct {
    ID    int64   `po:"id,primaryKey,bigint,identity"`
    Name  string  `po:"name,varchar(255),notNull"`
    Price float64 `po:"price,numeric(10,2),notNull"`
}

// Order: auto-generated by default, but you may supply your own.
type Order struct {
    ID          int    `po:"id,primaryKey,integer,identityByDefault"`
    OrderNumber string `po:"order_number,varchar(50),unique,notNull"`
    ProductID   int64  `po:"product_id,bigint,notNull"`

    Product *Product `po:"-,belongsTo,foreignKey(product_id),references(id)"`
}
```

Inserting works the same either way — leave the ID zero and read it back with `Returning`:

```go
inserted, err := builder.Insert[models.Product](qb).
    Values(newProduct).
    Returning(
        builder.Col[models.Product]("ID"),
        builder.Col[models.Product]("Name"),
        builder.Col[models.Product]("Price"),
    ).
    ExecReturning(ctx)
// inserted[0].ID is the database-assigned identity value
```

The program then selects, updates, and counts with the same `Col[T]` helpers — see [cmd/identity-example/main.go](cmd/identity-example/main.go).

<details>
<summary><strong>Generated SQL</strong></summary>

```sql
CREATE TABLE IF NOT EXISTS products (
    id bigint GENERATED ALWAYS AS IDENTITY,
    name varchar(255) NOT NULL,
    description text,
    price numeric(10,2) NOT NULL,
    in_stock boolean NOT NULL DEFAULT true,
    created_at timestamptz NOT NULL DEFAULT NOW(),
    updated_at timestamptz NOT NULL DEFAULT NOW(),
    CONSTRAINT products_pkey PRIMARY KEY (id)
);

CREATE TABLE IF NOT EXISTS orders (
    id integer GENERATED BY DEFAULT AS IDENTITY,
    order_number varchar(50) NOT NULL UNIQUE,
    product_id bigint NOT NULL,
    quantity integer NOT NULL,
    total_price numeric(10,2) NOT NULL,
    status varchar(20) NOT NULL DEFAULT 'pending',
    created_at timestamptz NOT NULL DEFAULT NOW(),
    CONSTRAINT orders_pkey PRIMARY KEY (id)
);
```

Reach for `bigint` + `identity` on anything high-volume; use `identityByDefault` only when you genuinely need to bring your own IDs.
