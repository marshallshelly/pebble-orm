---

You know the drill. Write the struct. Write the `CREATE TABLE` that matches it. Write the `SELECT`, the `Scan` with eleven `&` arguments, the slice append loop. Change one column and update all four places.

Pebble reads the struct tags and does the other three.

## Before / after

Fetching adults, newest first, with `database/sql`:

```go
rows, err := db.Query("SELECT id, name, email, age, created_at FROM users WHERE age >= $1 ORDER BY created_at DESC LIMIT 10", 18)
if err != nil { /* ... */ }
defer rows.Close()

var users []User
for rows.Next() {
    var u User
    if err := rows.Scan(&u.ID, &u.Name, &u.Email, &u.Age, &u.CreatedAt); err != nil { /* ... */ }
    users = append(users, u)
}
if err := rows.Err(); err != nil { /* ... */ }
```

With Pebble:

```go
users, err := builder.Select[User](qb).
    Where(builder.Gte(builder.Col[User]("Age"), 18)).
    OrderByDesc(builder.Col[User]("CreatedAt")).
    Limit(10).
    All(ctx)
// users is []User. Rename a column in the struct tag and Col[User] follows.
```

## How it works

One struct is the single source of truth for three things:

```
type User struct {
    ID    string `po:"id,primaryKey,uuid,default(gen_random_uuid())"`
    Email string `po:"email,varchar(320),unique,notNull"`
}
        â”‚
        â”śâ”€â–ş Query builder    Select[User] / Insert[User] / Update[User] / Delete[User]
        â”śâ”€â–ş Schema registry  column names, types, constraints, relationships
        â””â”€â–ş Migrations       pebble generate diffs structs vs DB â†’ timestamped up/down SQL
```

The CLI parses your `.go` files via AST â€” your models don't even have to compile for `pebble generate` to work, and no database connection is needed for the first migration (it replays your existing `*.up.sql` files as the baseline).

## Quick start

```go
package main

import (
    "context"
    "log"
    "time"

    "github.com/marshallshelly/pebble-orm/pkg/builder"
    "github.com/marshallshelly/pebble-orm/pkg/registry"
    "github.com/marshallshelly/pebble-orm/pkg/runtime"
)

// table_name: users
type User struct {
    ID        string    `po:"id,primaryKey,uuid,default(gen_random_uuid())"`
    Name      string    `po:"name,varchar(255),notNull"`
    Email     string    `po:"email,varchar(320),unique,notNull"`
    Age       int       `po:"age,integer,notNull"`
    CreatedAt time.Time `po:"created_at,timestamptz,default(NOW()),notNull"`
}

func main() {
    ctx := context.Background()

    if err := registry.Register(User{}); err != nil {
        log.Fatal(err)
    }

    db, err := runtime.ConnectWithURL(ctx, "postgres://localhost:5432/mydb?sslmode=disable")
    if err != nil {
        log.Fatal(err)
    }
    defer db.Close()

    qb := builder.New(db)

    users, err := builder.Select[User](qb).
        Where(builder.Gte(builder.Col[User]("Age"), 18)).
        Limit(10).
        All(ctx)
    if err != nil {
        log.Fatal(err)
    }

    for _, u := range users {
        log.Printf("%s <%s>", u.Name, u.Email)
    }
}
```

Then generate and apply the schema:

```bash
pebble generate --name initial_schema --models ./internal/models   # no DB needed
pebble migrate up --all --db "postgres://..."
```

For a production layout (`cmd/`, `internal/models/`, `internal/database/`), see [examples/basic](examples/basic/).

## Schema tags

Format: `po:"column_name,option1,option2(value)"`. First element is always the column name (`-` for relationship-only fields).

| Category | Options |
|----------|---------|
| Types | `uuid`, `varchar(n)`, `text`, `smallint`, `integer`, `bigint`, `numeric(p,s)`, `boolean`, `timestamp`, `timestamptz`, `jsonb`, `text[]`, `bytea`, `inet`, geometric types, â€¦ |
| Constraints | `primaryKey`, `notNull`, `unique`, `default(expr)` |
| Auto-increment | `serial`, `bigserial`, `identity`, `identityAlways`, `identityByDefault` |
| Foreign keys | `fk:table(column)`, `onDelete:CASCADE`, `onUpdate:SETNULL` |
| Indexes | `index`, `index(name)`, `index(name,gin)`, `index(name,btree,desc)` |
| Generated | `generated(expr)` + optional `virtual` |
| Enums | `enum(a,b,c)` â€” emits `CREATE TYPE ... AS ENUM` in migrations |
| Relationships | `belongsTo`, `hasOne`, `hasMany`, `manyToMany` + `foreignKey(...)`, `references(...)`, `joinTable(...)` |

Table-level directives live in comments above the struct:

```go
// table_name: users
// index: idx_active_users ON (email) WHERE deleted_at IS NULL
type User struct { ... }
```

## Query builder

```go
// SELECT with conditions, ordering, pagination
users, err := builder.Select[User](qb).
    Where(builder.Gt("age", 18)).
    And(builder.Like("email", "%@example.com")).
    OrderByDesc("created_at").
    Limit(10).Offset(20).
    All(ctx)

// First, Count, Exists
user, err  := builder.Select[User](qb).Where(builder.Eq("id", 1)).First(ctx)
count, err := builder.Select[User](qb).Where(builder.Gt("age", 21)).Count(ctx)

// INSERT â€” single, bulk, upsert, RETURNING
inserted, err := builder.Insert[User](qb).Values(u).Returning("*").ExecReturning(ctx)
n, err := builder.Insert[User](qb).Values(users...).Exec(ctx)
n, err  = builder.Insert[User](qb).Values(u).
    OnConflictDoUpdate([]string{"email"}, map[string]interface{}{"name": "Updated"}).
    Exec(ctx)

// UPDATE / DELETE
n, err = builder.Update[User](qb).Set("age", 31).Where(builder.Eq("id", 1)).Exec(ctx)
n, err = builder.Delete[User](qb).Where(builder.Lt("age", 18)).Exec(ctx)
```

## Relationships

```go
type Author struct {
    ID    int    `po:"id,primaryKey,serial"`
    Name  string `po:"name,varchar(100),notNull"`
    Books []Book `po:"-,hasMany,foreignKey(author_id),references(id)"`
}

// Eager loading â€” batched with ANY($1), no N+1
authors, err := builder.Select[Author](qb).Preload("Books").All(ctx)

// Nested preloads via dot notation
orders, err := builder.Select[Order](qb).Preload("Customer.Address").All(ctx)
```

All four shapes: `belongsTo`, `hasOne`, `hasMany`, `manyToMany` (junction table auto-named alphabetically: `roles_users`, not `users_roles`).

## Transactions

```go
if err != nil { return err }
defer tx.Rollback()

_, err = builder.TxInsert[User](tx).Values(user).ExecReturning()

account, err := builder.TxSelect[Account](tx).
    Where(builder.Eq("id", accountID)).
    ForUpdate().          // row lock
    First()

tx.Savepoint("before_update")
// ...
tx.RollbackToSavepoint("before_update")

return tx.Commit()
```

## Migrations

```bash
pebble generate --name add_users --models ./internal/models        # diff vs migration files (offline)
pebble generate --name add_users --models ./internal/models --db "postgres://..."   # diff vs live DB
pebble migrate up --all --db "postgres://..."
pebble migrate down --steps 1 --db "postgres://..."
pebble migrate status --db "postgres://..."
```

What the pipeline gets right so you don't have to:

- **Dependency ordering** â€” enums before tables, tables topologically sorted by FK references, indexes after columns
- **Serial semantics** â€” `serial` normalizes to `integer` + sequence when diffing, so it never generates bogus `ALTER TABLE ... TYPE serial`
- **Safe type changes** â€” automatic `USING` clauses for `text â†’ jsonb`, `text â†’ text[]`, `varchar â†’ integer`; a commented `MANUAL MIGRATION REQUIRED` block when no safe cast exists
- **Idempotent SQL** â€” `IF NOT EXISTS` on by default
- **Concurrency** â€” applies under a PostgreSQL advisory lock, tracked in `schema_migrations`
- **Down migrations** â€” every up file gets a generated reverse

## CLI

```bash
pebble generate --name NAME [--models DIR] [--db URL] [--empty]
pebble generate metadata --scan ./internal/models    # bake table names for production builds
pebble migrate up   [--all | --steps N] [--dry-run] [--interactive]
pebble migrate down [--steps N | --target VERSION] [--dry-run] [--interactive]
pebble migrate status [--json]
pebble introspect [--table TABLE] [--json]
pebble diff [--output FILE]
```

| Command | What it does |
|---------|--------------|
| `generate` | Diff structs against the DB (or migration files) â†’ timestamped `.up.sql` / `.down.sql` |
| `generate metadata` | Scan `// table_name:` comments â†’ `table_names.gen.go`, so custom names survive compiled builds |
| `migrate up/down/status` | Apply, roll back, inspect â€” flag-driven for CI, `-i` for a Bubbletea TUI |
| `introspect` | Dump live schema (columns, PKs, FKs, indexes, constraints) as tables or JSON |
| `diff` | Preview the SQL a `generate` would produce, without writing files |

Global flags: `--db`, `--migrations-dir` (default `./migrations`), `--verbose`, `--json`.

## PostgreSQL features

**JSONB**, three ways (pick the first unless you have a reason):

```go
type Product struct {
    ID    int         `po:"id,primaryKey,serial"`
    Attrs *Attributes `po:"attrs,jsonb"`  // 1. plain struct pointer â€” recommended
    // Attrs schema.JSONB                 // 2. map[string]any, schemaless
    // Attrs schema.JSONBStruct[Attributes] // 3. typed wrapper, legacy
}
```

**Arrays** â€” native `[]string`/`[]int64` etc. just work; `schema.StringArray`, `schema.Int32Array`, â€¦ add text-format (`{a,b,c}`) parsing for PgBouncer / `simple_protocol` mode, and scan correctly through the builders in every query exec mode.

**Enums** â€” `po:"status,enum(pending,active,completed)"` on a named string type generates the `CREATE TYPE` and uses it in the column. New values diff to `ALTER TYPE ... ADD VALUE`.

Also: UUID, identity columns (PostgreSQL 10+), generated columns, partial/covering/expression indexes, full-text search types, geometric types.

## FAQ

**Why not GORM?**
Pebble gives you `[]User` at compile time instead of reflection at runtime, and its migrations are generated from a diff instead of `AutoMigrate` guessing at your intent. If you like your ORM to surprise you, GORM is right there.

**Why pgx and not `database/sql`?**
`database/sql` is an abstraction over databases you're not using. Going straight to pgx means binary protocol, real PostgreSQL types, and one less layer to debug at 2am.

**Do my models need to compile for the CLI to work?**
No. `pebble generate` parses your source files with `go/ast`. Half-finished refactor, missing imports â€” the migration still generates.

**Does it work without a database connection?**
For generation, yes: with no `--db`, Pebble replays your existing `*.up.sql` files to reconstruct the baseline and diffs against that.

**PgBouncer?**
Yes â€” use the `schema.StringArray` family for array columns in `simple_protocol` mode, everything else just works.

**Does it scale?**
It's pgx with the boilerplate removed. Your indexes are the ceiling, not the ORM.
